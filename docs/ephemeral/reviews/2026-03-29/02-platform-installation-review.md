# Platform Compatibility & Installation Review

**Reviewer:** DevOps/Platform Engineering
**Date:** 2026-03-29
**Spec Version:** 1.2 (2026-03-28)
**Scope:** macOS X, Linux (Ubuntu 22.04+, Debian 12+) -- cross-platform installation, CLI, dependencies
**Documents Reviewed:** All 12 design subdocuments + master proposal

---

## Executive Assessment

The xpatcher design is architecturally sound for cross-platform use. The Python dispatcher, YAML artifacts, and `claude -p` subprocess model impose no inherent platform barriers. However, the `install.sh` script as written **will fail on macOS** due to a known `readlink -f` incompatibility, and several secondary issues (Python discovery, shell detection, dependency pinning, uninstall/upgrade paths) need resolution before the installer can be considered production-ready.

**Verdict:** 2 critical issues (blockers on macOS), 4 major issues, 7 minor issues. All are fixable within the Phase 5 packaging timeline. None require architectural changes.

## Critical Issues

### CRIT-01: `readlink -f` does not exist on macOS

**File:** `install.sh`, line 137 (Section 11, Implementation Roadmap)
**Platforms affected:** macOS (all versions)
**Impact:** The `bin/xpatcher` CLI wrapper script will fail immediately on every invocation.

The generated wrapper script contains:

```bash
SCRIPT_DIR="$(cd "$(dirname "$(readlink -f "$0")")/.." && pwd)"
```

On macOS, `readlink` is the BSD variant which does not support the `-f` (canonicalize) flag. Running this produces:

```
readlink: illegal option -- f
usage: readlink [-n] [file ...]
```

**Remediation options (choose one):**

**(A) POSIX-portable replacement (recommended):**
```bash
SCRIPT_DIR="$(cd "$(dirname "$0")/.." && pwd -P)"
```
This works on both macOS and Linux. It does not resolve symlinks in the script path itself, but since the wrapper is generated by the installer into a known location, this is acceptable. If symlink resolution is needed:

**(B) Portable `readlink -f` equivalent:**
```bash
_realpath() {
  local target="$1"
  cd "$(dirname "$target")" || return 1
  target="$(basename "$target")"
  while [ -L "$target" ]; do
    target="$(readlink "$target")"
    cd "$(dirname "$target")" || return 1
    target="$(basename "$target")"
  done
  echo "$(pwd -P)/$target"
}
SCRIPT_DIR="$(dirname "$(_realpath "$0")")"
```

**(C) Require `coreutils` on macOS:**
```bash
if command -v greadlink &>/dev/null; then
  SCRIPT_DIR="$(cd "$(dirname "$(greadlink -f "$0")")/.." && pwd)"
else
  SCRIPT_DIR="$(cd "$(dirname "$0")/.." && pwd -P)"
fi
```
This adds a Homebrew dependency (`brew install coreutils`) which is undesirable for a clean install story.

**Recommendation:** Option A. The wrapper script is generated into `$INSTALL_DIR/bin/` by the installer, so its location is deterministic. Symlink resolution of the script itself is unnecessary.

### CRIT-02: `python3 -m venv` fails on stock Ubuntu/Debian without `python3-venv`

**File:** `install.sh`, line 128
**Platforms affected:** Ubuntu 22.04+, Debian 12+
**Impact:** Installation fails with a confusing error about missing `ensurepip`.

On Ubuntu and Debian, `python3` is pre-installed but `python3 -m venv` requires the separate `python3-venv` package. Without it:

```
$ python3 -m venv /tmp/test
The virtual environment was not created successfully because ensurepip is not
available.  On Debian/Ubuntu systems, you need to install the python3-venv
package using the following command.

    apt install python3-venv
```

The installer must detect this and either:
1. Check for `python3 -m venv --help` success before attempting venv creation
2. Provide a clear error message directing the user to `sudo apt install python3-venv`
3. Or fall back to `python3 -m pip install --user` (less isolated but no extra package needed)

**Remediation:**

```bash
# After Python version check, verify venv module
if ! python3 -c "import venv" 2>/dev/null; then
    echo "ERROR: Python venv module not found."
    echo "  On Ubuntu/Debian: sudo apt install python3-venv"
    echo "  On macOS: venv is included with Python from Homebrew or python.org"
    exit 1
fi
```

## Major Issues

### MAJ-01: No Python version validation beyond existence check

**File:** `install.sh`, lines 101-106
**Platforms affected:** All

The installer checks that `python3` exists and prints the version, but never validates that it is >= 3.10. On Ubuntu 20.04 (still in LTS until 2025, widely used in CI), `python3` is 3.8. The install will succeed but xpatcher will crash at runtime due to `match` statements, `X | Y` union types, and other 3.10+ syntax.

```bash
# Current: just prints version, no validation
PY_VERSION=$(python3 -c "import sys; print(f'{sys.version_info.major}.{sys.version_info.minor}')")
echo "[OK] Python $PY_VERSION"
```

**Remediation:**

```bash
PY_VERSION=$(python3 -c "import sys; print(f'{sys.version_info.major}.{sys.version_info.minor}')")
PY_OK=$(python3 -c "import sys; print(1 if sys.version_info >= (3, 10) else 0)")
if [ "$PY_OK" != "1" ]; then
    echo "ERROR: Python 3.10+ required, found $PY_VERSION"
    exit 1
fi
echo "[OK] Python $PY_VERSION"
```

### MAJ-02: No fallback from `python3` to `python`

**File:** `install.sh`, line 101
**Platforms affected:** Some Linux minimal/container images, older systems

On some systems (notably some Docker base images, RHEL/CentOS derivatives, or Python installed from source without altinstall), the binary is `python` not `python3`. The installer should attempt both:

```bash
PYTHON=""
for candidate in python3 python; do
    if command -v "$candidate" &>/dev/null; then
        ver=$("$candidate" -c "import sys; print(sys.version_info >= (3, 10))" 2>/dev/null)
        if [ "$ver" = "True" ]; then
            PYTHON="$candidate"
            break
        fi
    fi
done
if [ -z "$PYTHON" ]; then
    echo "ERROR: Python 3.10+ not found. Tried: python3, python"
    exit 1
fi
```

Then use `$PYTHON` consistently instead of hardcoded `python3`.

### MAJ-03: Hook scripts use `python` instead of venv Python

**File:** `settings.json` (Section 7.4)
**Platforms affected:** All

```json
"command": "python .claude-plugin/hooks/pre_tool_use.py"
```

This invokes the system `python`, not `python3`, and not the venv Python. On macOS, `python` may not exist at all (Apple removed the `/usr/bin/python` symlink). On Linux, `python` might be Python 2. Even if `python3` is used, it would not have access to packages installed in the xpatcher venv (pydantic, pyyaml, rich).

**Current hooks only use stdlib** (`json`, `sys`, `os`, `pathlib`), so this works today. But if hooks ever import pydantic or pyyaml, they will fail.

**Remediation:** Use the venv Python explicitly:

```json
"command": "$XPATCHER_HOME/.venv/bin/python .claude-plugin/hooks/pre_tool_use.py"
```

Or, since settings.json may not support variable expansion, the install script should template this value at install time, or use a wrapper script.

### MAJ-04: No Claude Code CLI version check

**File:** `install.sh`, lines 109-112
**Platforms affected:** All

The installer verifies `claude` exists but not its version. xpatcher depends on specific CLI features:
- `--output-format json` / `--output-format stream-json`
- `--agent` flag (noted as potentially non-existent in Open Question #1)
- `--resume` for session continuation
- `--bare` mode
- `--max-turns`
- `--allowedTools`
- `--plugin-dir`

If the user has an older Claude Code CLI, xpatcher will fail with cryptic errors at runtime.

**Remediation:**

```bash
CLAUDE_VERSION=$(claude --version 2>/dev/null | head -1)
echo "[OK] Claude Code CLI: $CLAUDE_VERSION"
# TODO: parse and validate minimum version once version requirements are pinned
```

At minimum, the installer should capture and display the version. Ideally, a minimum version should be documented and enforced.

## Minor Issues

### MIN-01: Shell compatibility -- bashisms in install script

**File:** `install.sh`
**Platforms affected:** Systems where `/bin/sh` is dash (Ubuntu, Debian)

The script uses `#!/usr/bin/env bash` which is correct -- it will find bash. However, the PATH check on line 145 uses `[[ ]]` (bash-only) which is fine since bash is specified. No issue here, but worth noting:

- `set -euo pipefail` -- correct for bash, would fail in dash
- `&>/dev/null` -- bash-only, POSIX equivalent is `>/dev/null 2>&1`
- `[[ ":$PATH:" != *":$INSTALL_DIR/bin:"* ]]` -- bash pattern matching

**Assessment:** The shebang is bash-explicit, so these are acceptable. Just ensure the script is never accidentally invoked as `sh install.sh` (which would use dash on Ubuntu/Debian and fail). Add a guard:

```bash
if [ -z "$BASH_VERSION" ]; then
    echo "ERROR: This script requires bash. Run with: bash install.sh"
    exit 1
fi
```

### MIN-02: `cp -r` behavior differs between macOS and Linux

**File:** `install.sh`, lines 120-123
**Platforms affected:** macOS (minor)

```bash
cp -r .claude-plugin/ "$INSTALL_DIR/.claude-plugin/"
cp -r src/ "$INSTALL_DIR/src/"
```

On macOS, if `$INSTALL_DIR/.claude-plugin/` already exists, `cp -r` copies the *contents* into the existing directory, potentially leaving stale files from a previous install. On Linux with GNU coreutils, the behavior is the same but can result in `src/src/` nesting depending on trailing slash conventions.

**Remediation:** Delete the target directory first, or use `rsync`:

```bash
rm -rf "$INSTALL_DIR/.claude-plugin" "$INSTALL_DIR/src"
cp -r .claude-plugin "$INSTALL_DIR/.claude-plugin"
cp -r src "$INSTALL_DIR/src"
```

Note: remove trailing slashes from source paths for consistent behavior.

### MIN-03: No git version check for worktree support

**File:** Section 2.5 (System Architecture) references `git worktree`
**Platforms affected:** Older Linux systems

`git worktree` was introduced in git 2.5 (2015) with substantial improvements through 2.15 (2017). The `git worktree add -b` syntax used in the spec requires git >= 2.5. Ubuntu 22.04 ships git 2.34.1, Debian 12 ships 2.39.5, both well above the minimum. macOS Xcode CLT provides git 2.39+.

However, `git worktree list --porcelain` (likely needed for programmatic worktree management) requires git 2.8+.

**Remediation:** Add a minimum git version check (2.15 is a safe floor) to the preflight check:

```python
# In preflight_check()
git_version = subprocess.check_output(["git", "--version"]).decode().strip()
# Parse "git version 2.34.1" and compare
```

### MIN-04: `$HOME/xpatcher` path may conflict with user projects

**Platforms affected:** All

If a user has a project named `xpatcher` in their home directory, the default install path `~/xpatcher/` will collide. The `XPATCHER_HOME` override addresses this, but the default should be less collision-prone.

**Suggestion:** Consider `~/.xpatcher/` (dotfile convention for tools) or `~/.local/share/xpatcher/` (XDG convention on Linux). The current `~/xpatcher/` is visible and easy to find, which has usability benefits, so this is a trade-off.

### MIN-05: Case-insensitive filesystem (macOS default) may cause artifact collisions

**Platforms affected:** macOS with default APFS (case-insensitive)

Feature directories use kebab-case (e.g., `auth-redesign`). On macOS's default case-insensitive filesystem, `Auth-Redesign` and `auth-redesign` would refer to the same directory. Since feature slugs are generated from user input, two features whose slugs differ only by case would collide silently.

**Impact:** Low (unlikely in practice since slug generation should normalize to lowercase). Document the assumption that feature slugs are always lowercase.

### MIN-06: Emoji characters in install output

**File:** `install.sh`
**Platforms affected:** Systems with limited Unicode support, CI environments, older terminals

The installer uses `[OK]` (note: the actual spec uses a checkmark character). Some terminals, CI systems, and log aggregators do not render Unicode well.

**Remediation:** Use ASCII-safe indicators: `[OK]` or `[+]` instead of Unicode checkmarks.

### MIN-07: `python -m src.dispatcher.core` module resolution

**File:** `install.sh`, line 139 (the generated wrapper)
**Platforms affected:** All

```bash
exec "$SCRIPT_DIR/.venv/bin/python" -m src.dispatcher.core "$@"
```

This requires `src/` to be a proper Python package (with `__init__.py` at every level) and the working directory or `PYTHONPATH` to be set correctly. Running `python -m src.dispatcher.core` from `$INSTALL_DIR` requires `$INSTALL_DIR` to be the working directory or on `sys.path`.

The wrapper sets `XPATCHER_HOME` but does not `cd` to it or set `PYTHONPATH`. This will fail with `ModuleNotFoundError: No module named 'src'`.

**Remediation:**

```bash
#!/usr/bin/env bash
SCRIPT_DIR="$(cd "$(dirname "$0")/.." && pwd -P)"
export XPATCHER_HOME="$SCRIPT_DIR"
export PYTHONPATH="$SCRIPT_DIR:$PYTHONPATH"
exec "$SCRIPT_DIR/.venv/bin/python" -m src.dispatcher.core "$@"
```

Or, better yet, install xpatcher as a proper package in the venv:

```bash
"$INSTALL_DIR/.venv/bin/pip" install -e "$INSTALL_DIR"
```

This requires `pyproject.toml` to define a `[project.scripts]` entry point, which is more robust than shell wrappers and is the approach implied by Phase 5's `pip install xpatcher` goal.

## CLI Entry Point Analysis

The `bin/xpatcher` wrapper script (generated by the installer) has three concerns:

### 1. Path Resolution (CRIT-01 above)

Replace `readlink -f` with portable alternative.

### 2. Module Execution (MIN-07 above)

`python -m src.dispatcher.core` requires the install directory on PYTHONPATH.

### 3. Signal Handling

The wrapper uses `exec`, which is correct -- it replaces the bash process with the Python process, so signals (SIGINT, SIGTERM) are delivered directly to Python. This is the right approach for Ctrl+C handling during interactive TUI use.

### 4. Environment Inheritance

The wrapper exports `XPATCHER_HOME` but does not set or validate:
- `ANTHROPIC_API_KEY` (required by Claude Code CLI)
- `CLAUDE_CONFIG_DIR` (if user has custom Claude config)
- `PATH` (to ensure `claude` and `git` are findable from within the Python process)

**Recommendation:** The dispatcher's preflight check should validate that `claude` and `git` are on PATH from the Python process's perspective, not just from the shell.

## Missing: Uninstall/Upgrade Story

The spec has no uninstall or upgrade mechanism. This needs to be addressed for production packaging.

### Uninstall

Currently, uninstalling requires the user to know the installation layout:

```bash
rm -rf ~/xpatcher
# Remove PATH entry from .bashrc/.zshrc manually
```

**Recommendation:** Provide `xpatcher uninstall` or a separate `uninstall.sh`:

```bash
#!/usr/bin/env bash
INSTALL_DIR="${XPATCHER_HOME:-$HOME/xpatcher}"
echo "Removing xpatcher from $INSTALL_DIR"
rm -rf "$INSTALL_DIR"
echo "Done. Remove the PATH entry from your shell profile manually."
echo "  grep -n 'xpatcher' ~/.bashrc ~/.zshrc 2>/dev/null"
```

Note: Project-level `.xpatcher/` directories are **not** cleaned up by uninstall. This is correct -- those are project artifacts and the user may want to keep them.

### Upgrade

The `install.sh` script can be re-run (it uses `mkdir -p` and `cp -r`), but:

1. It does not clean up stale files from previous versions (MIN-02)
2. It recreates the venv only if `.venv` does not exist, so dependency updates are not applied on reinstall
3. There is no version tracking to tell the user what version they have

**Recommendation:**

1. Track installed version in `$INSTALL_DIR/VERSION`
2. On reinstall, compare versions and show a changelog
3. Force venv dependency refresh: `pip install --upgrade pydantic pyyaml rich`
4. Clean stale files before copying new ones

Long-term (Phase 5), `pip install --upgrade xpatcher` via PyPI is the cleanest upgrade path.

## Questions for Product Owner

1. **Install path convention:** `~/xpatcher/` is visible in the home directory. Should the default be `~/.xpatcher/` (hidden, XDG-ish) or stay visible? Tradeoff: discoverability vs home directory clutter.

2. **Minimum Claude Code CLI version:** What is the oldest Claude Code CLI version xpatcher will support? This determines which `--agent`, `--resume`, and `--output-format` flags can be relied upon. This must be pinned before Phase 5.

3. **Ubuntu 20.04 support:** Ubuntu 20.04 has Python 3.8 and is still in extended LTS. Is it explicitly out of scope, or should the docs/installer say so?

4. **Docker/CI usage:** Is running xpatcher in Docker or CI an intended use case? If so, the installation story needs a `Dockerfile` and/or GitHub Action. The current installer assumes an interactive shell (PATH guidance). The `settings.json` hook paths assume a specific filesystem layout.

5. **Multi-user server install:** `/opt/xpatcher/` is mentioned as a server-wide path. Who owns the venv? How do different users' configs interact? The venv is per-installation, not per-user, which means all users share dependency versions -- usually fine, but permissions need consideration.

6. **pip install timeline:** Phase 5 mentions `pip install xpatcher`. Is `pyproject.toml` complete enough today? The directory layout (`src/dispatcher/`) should use `src/xpatcher/` for proper package naming. This affects the module path in the wrapper script and import structure.
